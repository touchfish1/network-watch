# 执行计划（2026-04-15）

## 目标

围绕“可回看 + 多主机易用”继续迭代，形成一套可持续执行的版本计划：

1. 事件流可回看能力完整（时间窗、分页、筛选、状态反馈）。
2. 历史趋势支持单机与全局聚合两种查看模式，并保证性能可接受。
3. 悬窗与 Web 在事件/历史体验上保持一致，避免两套逻辑割裂。

## 范围与约束

- 后端沿用 `SQLite(rusqlite)`，不引入外部数据库。
- 接口维持 `v1` 路径，优先向后兼容。
- 每个阶段都必须通过：
  - `cargo check --features "desktop,agent"`
  - `npm run build`

## 分阶段计划

### Phase 1：事件流回看基础（已完成）

- [x] 后端事件查询支持 `since_ms/until_ms/offset/limit`。
- [x] Web 事件流支持 `1h/24h/7d` 时间窗。
- [x] Web 事件流支持上一页/下一页分页。
- [x] Web 事件流支持当前主机/全部主机作用域切换。
- [x] Web 事件流支持类型与关键词过滤。
- [x] 空页自动回退、分页按钮禁用状态、回到最新按钮、页码显示。

验收标准：

- 在不同时间窗与过滤条件下分页切换无报错。
- 翻到空页可自动回退到最近有数据页。
- 处于最新页时“最新/上一页”按钮状态正确。

### Phase 2：历史趋势体验增强（进行中）

- [x] 新增 `/api/v1/history/aggregate` 聚合接口。
- [x] Web 历史趋势支持当前主机 / 全部主机聚合切换。
- [x] 全部主机聚合增加缓存与请求去重。
- [x] 历史趋势增加“刷新中/无数据”可视状态。
- [x] 聚合曲线增加点位信息（时间与值，悬停提示）。

验收标准：

- 主机数增加时页面仍可在可接受时间内完成刷新。
- 切换范围 `24h/7d` 时数据和文案一致。

### Phase 3：悬窗事件中心（进行中）

- [x] 悬窗事件卡增加分页与时间窗筛选。
- [x] 悬窗事件卡增加“仅当前主机 / 全部主机”切换。

## 执行节奏

- 每次迭代按“文档更新 -> 代码实现 -> 构建验证 -> 日志回写”闭环执行。
- 若阶段中途变更范围，先更新本文件再改代码。

## 风险与对策

- 风险：事件过滤在前端做，可能导致“分页后再过滤”数量不稳定。  
  对策：下一阶段可下沉事件类型/关键词过滤到后端查询参数。

- 风险：全局聚合在主机规模增大时可能变慢。  
  对策：继续保留缓存，并在必要时增加预聚合表或后台定时聚合。
