# 跨平台状态栏系统监控器项目计划

## Summary

项目按“纯原生分平台 UI + C++ 共享核心”的方式设计，首版目标为“监控 + 告警版”。产品以系统状态栏/托盘常驻应用为中心，实时展示网络、CPU、内存概览；点击图标打开详细监控窗口；支持阈值告警、桌面通知和短期历史趋势。

实现上采用单仓多模块结构：共享 C++ 核心负责采集、聚合、存储、告警与配置；Linux、macOS、Windows 各自实现原生托盘、窗口、通知和平台接入层。这样既满足 C++ 工程诉求，也兼顾三端原生体验。

## Key Changes

### 1. 架构与模块划分

- 构建系统使用 `CMake`，统一管理共享核心与三端目标产物。
- 建立共享核心模块：
  - `metrics-core`：采集调度、数据模型、速率计算、平滑处理、历史缓存。
  - `alert-core`：阈值规则、触发抖动保护、冷却时间、告警事件总线。
  - `config-core`：配置读写、默认值、平台路径适配。
- 建立平台 UI 壳层：
  - Windows：Win32/WinUI 托盘 + 原生窗口 + Toast 通知。
  - macOS：AppKit 状态栏项 + 原生监控窗口 + 用户通知；必要处允许 Objective-C++ 作为桥接层。
  - Linux：GTK/libappindicator 或 StatusNotifierItem 托盘 + GTK 窗口 + `libnotify` 通知。
- 明确线程模型：
  - 后台采集线程周期拉取指标。
  - UI 主线程只订阅聚合结果，不直接采集。
  - 告警引擎独立消费采样结果，避免 UI 卡顿影响告警。

### 2. 功能范围

- 状态栏/托盘展示：
  - 默认显示网络上下行速率。
  - 悬停或菜单显示 CPU、内存、网络摘要。
  - 支持低频刷新文本，高频采样后做 UI 降采样。
- 监控窗口：
  - 总览卡片：CPU 占用、内存占用、网络上下行。
  - 趋势图：最近 1 分钟、5 分钟、30 分钟。
  - 网卡页：列出接口名、连接状态、当前速率、IP 基本信息。
- 告警能力：
  - CPU 持续高占用。
  - 内存占用超过阈值。
  - 网络吞吐持续过高或掉到 0。
  - 每类规则支持阈值、持续时间、冷却时间、启用开关。
- 配置能力：
  - 刷新间隔、显示模式、启动自启、告警阈值、通知开关。
  - 本地持久化，按平台放在用户配置目录。

### 3. 共享接口与数据模型

- 核心统一暴露以下接口概念：
  - `MetricSample`：单次采样快照，含时间戳、CPU、内存、网络总量与网卡列表。
  - `MetricDelta`：基于相邻样本计算出的速率与变化值。
  - `TraySummary`：给托盘层消费的轻量文本/图标状态模型。
  - `AlertRule`：规则定义，含指标类型、阈值、持续时间、冷却时间。
  - `AlertEvent`：触发/恢复事件，供通知层和日志层消费。
  - `Settings`：全局配置对象。
- 平台适配层统一实现：
  - `IMetricsProvider`
  - `ITrayAdapter`
  - `INotificationAdapter`
  - `IAutostartAdapter`
- 指标口径统一：
  - CPU：整机总占用百分比。
  - 内存：已用/总量与百分比。
  - 网络：系统级总上下行速率，并补充网卡级明细。
- 历史缓存先采用内存环形缓冲区，首版不引入数据库；仅在后续扩展长期历史时再加持久化时序存储。

### 4. 平台实现策略

- Windows：
  - 通过 PDH / IP Helper API / `GlobalMemoryStatusEx` 获取指标。
  - 托盘图标支持动态文本或状态图标切换。
  - 开机启动通过注册表或 Startup 快捷方式实现。
- macOS：
  - 通过 `host_statistics`、`sysctl`、`getifaddrs` 等接口获取指标。
  - 状态栏使用 `NSStatusItem`，窗口使用 AppKit。
  - 自启动通过 LaunchAgent。
- Linux：
  - 优先读取 `/proc/stat`、`/proc/meminfo`、`/proc/net/dev`。
  - 托盘优先兼容 AppIndicator/StatusNotifier，必要时准备降级路径。
  - 自启动通过 `.desktop` autostart。
- 平台差异在适配层解决，核心不直接引用平台 UI API。

### 5. 交付节奏

- 第 1 阶段：仓库初始化、CMake 工程骨架、核心接口、平台壳层空实现。
- 第 2 阶段：CPU/内存/网络采集打通，托盘实时展示。
- 第 3 阶段：详细监控窗口与历史趋势。
- 第 4 阶段：告警规则、通知、自启动、设置页。
- 第 5 阶段：打包分发、跨平台验收、性能与稳定性优化。

## Test Plan

- 单元测试：
  - 速率计算、采样窗口、异常值平滑、阈值判定、冷却时间逻辑。
  - 配置解析与默认值回退。
- 集成测试：
  - 模拟采样流驱动告警引擎，验证触发与恢复事件。
  - 托盘摘要模型在不同刷新频率和极端值下输出正确。
- 平台验证：
  - Windows/macOS/Linux 各自验证托盘常驻、点击打开窗口、通知弹出、自启动成功。
  - 多网卡场景、断网重连、睡眠唤醒后恢复采集。
- 性能与稳定性：
  - 常驻 24 小时内内存无明显泄漏。
  - 默认采样频率下 CPU 占用保持低位。
  - 高网络吞吐时 UI 不抖动、不阻塞。
- 验收场景：
  - 用户能在托盘直接看到网络速率。
  - 点击后可查看 CPU/内存/网络趋势。
  - 超阈值后在设定持续时间内触发通知，冷却期内不重复轰炸。

## Assumptions

- 项目语言主体为 C++，但 macOS 原生状态栏接入允许使用少量 Objective-C++ 桥接，这是纯原生方案下的必要默认。
- 首版采用“C++ 共享核心 + 三端原生 UI 壳层”，不追求单一 UI 代码跨平台复用。
- 历史数据仅保留短期本地内存缓存，不做长期存储、云同步或远程监控。
- 告警范围默认覆盖 CPU、内存、网络三类系统级指标；不做进程级监控和网络诊断工具。
- 文档产出默认包含：

  - `docs/architecture.md`：架构设计
  - `docs/roadmap.md`：阶段计划与验收标准
